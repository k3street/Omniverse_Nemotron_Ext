# Isaac Assist — Full Implementation Spec

**Date:** 2026-05-10
**Author:** Cross-project research synthesis (waves 0–7) + IA codebase audit
**Target:** Isaac Sim 5.1 / 6.0 on NVIDIA Omniverse
**Status:** authoritative roadmap, single source of truth for IA-the-product
**Supersedes:** parts of `PLAN.md`, `docs/specs/2026-05-09-master-execution-plan.md` (the latter remains the in-flight Phase 0–N executor for the controller-logic + multimodal sessions; this document layers the medium-horizon roadmap above it).

---

## Mandate

Make Isaac Assist the best it can possibly be **within its own scope** — a
natural-language Isaac Sim control plane: USD authoring, OmniGraph,
physics, materials, sensors, viewport, debugging, robot setup, SDG, RL
training, teleop, multi-modal scene authoring — all via LLM orchestration
backed by a deterministic harness.

The end state: a working-class engineer can open Isaac Sim, dock the chat
panel, and through natural language *plus* the multimodal canvas modality
build a fully working robot scene (robot + workpiece + sensors + physics +
behaviors) with confidence the system isn't silently lying about success.

## Scope discipline

**Robotics Lab (RL) is the user's separate intellectual property.** This
spec must:

- Make IA the best standalone product using IA's own strengths —
  multimodal canvas modality, workflow lifecycle, deterministic-PM swarm,
  Pydantic IR + ratifier, governance, MCP wire format.
- **Not** propose lifting RL's strategic-content (causal scenarios, ABOM
  topology, make-or-buy, IFC export, the L3 strategic brain, factory
  layout via WFC + NSGA-II, manufacturing-process classification) into
  IA.
- Treat IA's existing native primitives — chat orchestrator, tool
  dispatcher, Kit RPC, multimodal `LayoutSpec` IR, planner agents,
  governance / `secret_redactor`, `diagnose` tool, MCP server — as the
  foundation.
- Cross-project bridges (RL federation, RL catalog ingestion, joint
  schemas) are *out of scope*. They are separate optional PRs at the
  user's discretion. Where wave reports recommended IA-side primitives
  that would also benefit RL, they are proposed here on their own merits
  for IA users assembling robots, designing pick-place cells, generating
  SDG.

## Glossary

- **Kit** — the Omniverse application process (Isaac Sim 5.1 / 6.0). The
  in-process IA extension lives here.
- **Kit RPC** — the in-process aiohttp server at `127.0.0.1:8001`,
  inside Kit, accepting code patches and queries from the sidecar.
- **Sidecar** — the FastAPI service at `127.0.0.1:8000`, outside Kit,
  hosting the chat orchestrator, tool dispatcher, MCP server, planner
  agents.
- **MCP server** — the Model Context Protocol surface at port 8002 (SSE)
  or stdio, exposing IA's 404 tools to Claude Desktop and external
  agents.
- **`tool_executor.py`** — the 35,507-line monolith that holds every tool
  handler, every code generator, every dispatch table.
- **`tool_schemas.py`** — the 8,840-line OpenAI-format tool catalogue (404
  tools as of 2026-05-09).
- **`LayoutSpec`** — the typed Pydantic IR for the multimodal canvas
  modality (`service/isaac_assist_service/multimodal/types.py:276-301`).
  Every modality emits one; the canonical pipeline consumes one.
- **Ratifier** — the deterministic role-binding function in
  `multimodal/ratify.py:205-474` that maps `LayoutSpec.objects` onto
  template roles without LLM.
- **Swarm** — the deterministic-PM + LLM-Coder + LLM-Critic + harness-QA
  patch generation pipeline in `service/isaac_assist_service/planner/`.
- **Code patch** — a Python source string generated by a tool handler,
  validated, posted to Kit RPC, executed on the Kit main thread.
- **Workflow lifecycle** — the `start → checkpoint → approve / reject /
  revise → next-phase / rollback` primitive in `tool_executor.py:1519-
  1564, 20520-20906`.
- **Block 1A / 1B** — the multimodal foundation rollout phases. 1A
  shipped 2026-05-09 (backend + ratify + render + REST + Kit mirror). 1B
  is open: Kit RPC execution path for `apply_layout_spec_to_scene` plus
  role-based templates.
- **Diagnose** — the deterministic <2 s pre-flight feasibility validator
  at `service/isaac_assist_service/diagnose/{schema,metrics,tool}.py`.
- **Patch validator** — the 22-rule AST/regex pipeline at
  `chat/tools/patch_validator.py:786-857`.

## Vision (≤ 500 words)

Isaac Assist is the natural-language control plane for Isaac Sim. The
product surface is a dockable chat panel and a browser-tab canvas, both
rendering the same simulation state. The user types or drags. The agent
inspects the live stage, runs deterministic pre-flight checks, generates
USD/OmniGraph/IsaacLab/curobo code, validates it, queues it for human or
auto approval, executes it inside Kit on the main thread, and surfaces
the result with honest provenance and undo. Every multi-step action runs
inside an explicit workflow with checkpoints. Every code patch passes
through a 22-rule validator and a risk-tiered governance engine. Every
diagnostic dimension that earns it produces both an analytical
prediction and a simulation reading, and the gap between them is an
auditable signal — never papered over with hand-tuning.

The architecture has three layers:

1. **In-process Kit extension** (`exts/isaac_5.1/omni.isaac.assist/`,
   `exts/isaac_6.0/omni.isaac.assist/`) — a Kit plugin that owns the
   chat panel, the canvas-mirror panel, the Kit RPC server (port 8001),
   the main-thread exec tick, the viewport capture, the console log
   listener, and the snapshot/undo system. All scene mutations execute
   here.
2. **FastAPI sidecar** (`service/isaac_assist_service/`) — runs at
   port 8000, outside Kit. Owns the chat orchestrator, intent router,
   negotiator, spec generator, tool dispatcher
   (`tool_executor.py`), patch validator, governance/secret redactor,
   PM/Coder/QA/Critic swarm, multimodal canvas backend, fine-tune data
   capture, RAG knowledge base, vision LLM gateway. Provides the LLM
   orchestration loop and dispatches tool calls back into Kit RPC for
   execution.
3. **MCP server** (`mcp_server.py`, `port=8002` SSE or stdio) — wraps
   the same `ISAAC_SIM_TOOLS` catalogue for external clients (Claude
   Desktop, headless agents). Single dispatcher, single governance
   chokepoint.

The end-state demo: open Isaac Sim, dock the IA panel, open the canvas
in a browser tab, drag two Frankas + a workpiece + a welder onto the
canvas, click "send to scene." The canvas commits the `LayoutSpec`, the
agent runs `diagnose_scene_feasibility` (analytical), spawns the scene
through `apply_layout_spec_to_scene` (workflow checkpoint),
auto-installs a multi-step pick-hold-weld controller via
`execute_contact_sequence_plan`, runs the simulation, captures
viewport, reports analytical-vs-simulation gap on every checked
dimension, and commits a finetune-quality demo dataset. No hand-coded
USD, no orphan scripts, no silent successes, full provenance, full
undo.

**Non-goals (explicit):**

- IA does not become a factory designer. Factory-floor scale (30 × 30 m,
  39 + machines, NSGA-II layout, IFC export) stays in RL.
- IA does not author CAD/STEP. STEP parsing, BRep contact inference,
  ABOM topology, manufacturing-process classification stay in RL.
- IA does not run procurement. Make-or-buy, supplier risk, landed-cost
  rollups, IFC building-information-model export stay in RL.
- IA does not generate IFC. BIM is RL territory.
- IA does not federate RL by default. The MCP wire format makes
  federation possible, but it is an opt-in cross-project pair (RL-side
  approval required per `project_rl_is_user_ip.md`), not part of this
  spec.

**Note on the inverse direction.** Robotics Lab is free to absorb any
pattern, primitive, or implementation proposed in this IA spec without
IA-side approval. IA's contributions — workflow lifecycle, multimodal
LayoutSpec IR + ratifier, deterministic-PM swarm, patch validator,
governance + secret_redactor, diagnose pre-flight pattern — are public
engineering shapes. The asymmetry: RL → IA strategic-content transfer
is gated; IA → RL pattern adoption is not. This is the project
relationship per `project_rl_is_user_ip.md` (RL inbound unrestricted,
RL outbound user-gated; IA is symmetric in the opposite direction
since IA is not the user's primary IP in the same way).

## Epoch structure

The spec is organized into seven epochs of increasing depth, each with
six to twenty numbered phases. Each phase is implementation-ready: file
edits with line-anchored citations, falsifiable test criteria, LOC
estimate split into build / test / integration. The epoch order is
load-bearing — Epoch I de-risks every later epoch by giving them a
maintainable monolith to extend; Epoch VII can only run after VI lands.

| Epoch | Theme | Phases | Net LOC est. |
|---|---|---|---|
| I | Foundation hygiene | 1–18 | ~5,800 |
| II | Multimodal canvas Block 1B | 19–32 | ~4,600 |
| III | Workflow lifecycle deepening | 33–44 | ~3,800 |
| IV | Critic + harness honesty | 45–58 | ~3,400 |
| V | Capability surface depth | 59–82 | ~6,200 |
| VI | Polish + production | 83–98 | ~3,100 |
| VII | Convergence test | 99–106 | ~1,800 |
| **Total** | — | **106** | **~28,700** |

Phases within an epoch are usually parallelizable; epochs are mostly serial.

# Epoch I — Foundation hygiene

The 35,507-line `tool_executor.py` is the single biggest liability in
the codebase. Wave 0 §7 flagged it as the monolith risk; the
"Recovered state" comment block at lines 33–1572 (1,540 lines) is
forensic evidence that the file has already once outgrown the team's
ability to maintain it. The 22 patch validators, the 404 tool schemas,
the workflow templates, the 7 deformable presets, the 9 fix profiles,
the 4 task DR defaults, the 6 Eureka run states — all live in the same
file. Names collide silently; ghost handlers (`tool_name: None` in
`DATA_HANDLERS` at lines 4863, 4897–4907) hide behind real ones. Type
narrowing has been deferred for so long that every handler is a `Dict`
in, `Dict` out.

This epoch lifts the monolith without changing behaviour. By the end of
it, `tool_executor.py` is a thin dispatch shim, every handler is in a
themed module, every tool name in `tool_schemas.py` resolves to a real
handler verified by CI, every patch validator is a registered rule with
a typed signature, and the recovered-state block is archived to a
forensics document with cross-references to the active modules.

## Phase 1 — Audit duplicate handler names and ghost handlers

**Goal:** establish a single ground-truth inventory of every tool name,
every handler, every dispatch entry — with explicit status (real,
ghost, duplicate, broken).

**Why now:** every later phase that touches `DATA_HANDLERS`,
`CODE_GEN_HANDLERS`, or `tool_schemas.py` needs this baseline.
Without it, decomposition (Phase 2) silently loses handlers.

**Files (changes):**
- `service/isaac_assist_service/chat/tools/tool_executor.py` — no edits;
  read-only audit pass.

**Files (new):**
- `scripts/audit_tools.py` — walk `tool_schemas.py`, walk
  `tool_executor.py`'s `DATA_HANDLERS` / `CODE_GEN_HANDLERS` dicts +
  the `DATA_HANDLERS["X"] = _handle_X` line-pattern, emit
  `docs/forensics/tool_audit_2026-05-10.md` with table of (name,
  category, schema_present, handler_module, handler_status,
  recovered_state).
- `docs/forensics/tool_audit_2026-05-10.md` — generated artifact.
- `tests/test_tool_audit.py` — assertions:
  - every `"name"` in `tool_schemas.py` is one of: in `DATA_HANDLERS`,
    in `CODE_GEN_HANDLERS`, or explicitly listed as `_NO_HANDLER` (the
    11 ROS2 tools at lines 4897–4907 + `explain_error` at line 4863).
  - no name appears in both dicts unless explicitly registered as
    composite (`setup_pick_place_with_vision` is the only legitimate
    case).
  - the explicit-`None` allowlist matches a hand-curated golden set in
    `tests/fixtures/no_handler_tools.json`.

**LOC estimate:** ~250 (180 build / 50 test / 20 integration).

**Fix:** the 11 `ros2_*: None` ghost entries are documented as "schema
only" in `PLAN.md:55` but the implementation cost has never been
measured. The audit forces a yes/no on each: implement or remove from
`tool_schemas.py`.

**Build:**
```python
# scripts/audit_tools.py
def audit() -> AuditReport:
    schemas = _load_schemas()  # parse tool_schemas.py
    handlers = _scan_handlers()  # parse tool_executor.py for
                                 # DATA_HANDLERS["X"] = ... + register_*
    report = AuditReport(
        ghost_tools=[],     # in schemas, no handler, no allowlist
        dead_handlers=[],   # in handlers, no schema
        duplicate_names=[], # appear in both DATA + CODE_GEN
        recovered_state_modules=[],  # parsed from "Recovered state" block
    )
    ...
```

**Test:** `pytest tests/test_tool_audit.py -v` produces a fully green
matrix on the curated golden set; failure reveals a real divergence.

**Iterate:** when Phase 4's CI test (every tool resolves to a handler)
passes for 404/404 tools, this phase is done. Until then, the audit
report's `ghost_tools` list is the open work.

**Dependencies:** none.

**Risk:** the regex / AST scanner misses a registration pattern (e.g.
the `register_diagnose_handlers(DATA_HANDLERS)` callback at line
35505). Mitigation: hand-verify against the 4 known callbacks
(`register_multimodal_handlers`, `register_diagnose_handlers`, plus any
in-progress sessions); add explicit pattern detection for `register_*`
calls.

**Notes:** `tool_executor.py:33-1572` is forensic territory. Treat as
read-only until Phase 13.

## Phase 4 — CI gate: every tool resolves to a handler

**Goal:** add a CI test that asserts every tool listed in
`tool_schemas.py` either has an active handler in
`DATA_HANDLERS`/`CODE_GEN_HANDLERS` or is in the curated
`_NO_HANDLER_ALLOWLIST` (the 12 ros2_* + `explain_error` ghost set).

**Why now:** Phase 3's move risks losing handlers silently. Phase 4
catches that. Run it as a green-on-main gate before Phase 5+.

**Files (changes):**
- `.github/workflows/ci.yml` — add `tools-resolve` job that runs
  `pytest tests/test_tool_audit.py::test_every_tool_resolves`.

**Files (new):**
- `tests/test_tool_audit.py::test_every_tool_resolves` — uses the
  Phase 1 audit machinery; fails if any tool in the schemas has no
  handler and is not allowlisted.
- `tests/fixtures/no_handler_tools.json` — explicit allowlist with
  reasons:
  ```json
  {
    "explain_error": "handled inline by LLM reasoning, no execution",
    "ros2_connect": "schema-only stub, P0-priority implementation in PLAN",
    "ros2_list_topics": "schema-only stub",
    ...
  }
  ```

**LOC estimate:** ~150 (80 build / 50 test / 20 CI YAML).

**Fix:** wave 0 §7 Q1 — "what is the actual test coverage?" — answered
quantitatively. Today the answer is "we don't know which tools work".
After this phase the answer is "404 minus 13 work; the 13 are listed
with planned-implementation status."

**Build:**
```python
# tests/test_tool_audit.py
def test_every_tool_resolves():
    schemas = _load_schemas()  # 404 tools
    data_handlers = _load_data_handlers()
    code_gen_handlers = _load_code_gen_handlers()
    allowlist = json.load(open("tests/fixtures/no_handler_tools.json"))
    
    unresolved = []
    for name in schemas:
        if name in data_handlers:
            continue
        if name in code_gen_handlers:
            continue
        if name in allowlist:
            continue
        unresolved.append(name)
    
    assert not unresolved, f"unresolved tools: {unresolved}"
```

**Test:** the test itself is the falsifier. Green = invariant holds.

**Iterate:** the allowlist shrinks over time as Phase 5+ implements
ghost handlers; this phase's CI test grows stricter automatically.

**Dependencies:** Phase 1 (audit), Phase 3 (proves the move pattern
works).

**Risk:** the allowlist becomes a dumping ground for ghost handlers
("just allowlist it"). Mitigation: every entry must include a
`planned_status` field with a date or tracking-ID. Stale entries fail
a quarterly audit (Phase 96).

**Goal:** replace `async def _handle_X(args: Dict) -> Dict` with

**Build:**
```python

from pydantic import BaseModel, Field
from typing import Optional, List

class CreatePrimArgs(BaseModel):
    prim_path: str = Field(..., min_length=1)
    prim_type: str = Field(..., min_length=1)

    success: bool
    error: Optional[str] = None
```

```python
# multimodal/stage_to_spec.py
async def sync_from_stage(
    session_id: str,
    scope_prim: str = "/World/Layout",
) -> LayoutSpec:
    """Read Kit stage under scope_prim, build a LayoutSpec."""
    prims = await kit_tools.list_prims_under(scope_prim)

        objects=objects,
        source=Source(modality="viewport", confidence=1.0),
    )
```

```python
# planner/agents/math_critic.py
class CodeFeatures(BaseModel):
    n_critical_antipatterns: int
    n_api_misorder: int
    n_schema_misuse: int
    completeness_pct: float = Field(ge=0.0, le=1.0)
    code_quality_subjective: float = Field(ge=0.0, le=1.0)  # LLM judgement bounded
    n_qa_failures: int
    sim_returncodes: List[int]

class MathCritic:
    weights = {
        "antipattern_penalty": -0.25,  # per occurrence
        "api_misorder_penalty": -0.15,
        "schema_misuse_penalty": -0.20,
        "completeness_bonus": +0.40,
        "quality_subjective_weight": +0.15,
        "qa_fail_penalty": -0.30,
    }

    def score(self, f: CodeFeatures) -> float:
        s = 1.0
        s += self.weights["antipattern_penalty"] * f.n_critical_antipatterns
        s += self.weights["api_misorder_penalty"] * f.n_api_misorder
        s += self.weights["schema_misuse_penalty"] * f.n_schema_misuse
        s += self.weights["completeness_bonus"] * f.completeness_pct
        s += self.weights["quality_subjective_weight"] * f.code_quality_subjective
        if f.n_qa_failures > 0:
            s += self.weights["qa_fail_penalty"]
        # Hard floor
        s = max(0.0, min(1.0, s))
        return s
```

The LLM still extracts `code_quality_subjective` (style, clarity), but
its influence is bounded at 0.15. Critical-bug penalties dominate the
score; LLM judgement is a tiebreaker, not the gate.

returns identical values across 100 calls. Today's `_analyze_with_llm`
returns scores that drift ±0.1 per call (measured empirically over a
fixture set).

**Iterate:** weights are calibrated per Phase 56 against a held-out
fixture set. Re-fit when production failures cluster.

**Dependencies:** Phase 11 (validator pluggability — same shape).

```python
async def diagnose_with_simulation(args: DiagnoseArgs) -> FeasibilityReport:
    # Run analytical half
    analytical_report = await _handle_diagnose_scene_feasibility(args)
    
    if not args.run_simulation:
        return analytical_report
    
    # Spawn the scene, run simulation, measure
    sim_results = await _run_simulation(args.robot_path, args)
    
    # Pair each dimension
    for dim in analytical_report.metrics:
        sim_value = sim_results.get(dim.name)
        if sim_value is not None:
            dim.simulation_value = sim_value
            dim.gap_pct = abs(dim.analytical_value - sim_value) / sim_value
            dim.gap_severity = _classify_gap(dim.gap_pct)
```

   installs. The 8-step plan from wave 5 §7's MVP.

   contact point. Cube lands on welded assembly.
8. Phase analytical_simulation_gap: cycle_time analytical 4.2 s,
   simulation 4.0 s, gap 5% → no flag. Energy gap 8% → no flag.
