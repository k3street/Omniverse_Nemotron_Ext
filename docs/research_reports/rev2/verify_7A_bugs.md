```python
"""IsaacLab RL environment: {task}
Auto-generated by Isaac Assist.
"""
import isaaclab.envs.mdp as mdp
from isaaclab.envs import ManagerBasedRLEnvCfg
from isaaclab.managers import (
    ObservationGroupCfg,
    ObservationTermCfg as ObsTerm,
    RewardTermCfg as RewTerm,
    SceneEntityCfg,
)
from isaaclab.scene import InteractiveSceneCfg


class {task}EnvCfg(ManagerBasedRLEnvCfg):
    """Configuration for {task} environment."""

    # Scene
    scene = InteractiveSceneCfg(
        num_envs={num_envs},
        env_spacing={spacing},
    )

    # Observations
    observations = ObservationGroupCfg(
        "{o}": ObsTerm(func=mdp.{o}),  # one line per obs term
    )

    # Actions
    actions = mdp.{acts}   # e.g. mdp.joint_positions

    # Rewards
    rewards = {
        "{r}": RewTerm(func=mdp.{r}, weight=1.0),  # one per reward term
    }

    # Episode
    episode_length_s = {ep_len} * {decimation} / 120.0
    decimation = {decimation}
```

## Claim-by-Claim Verdicts

---

### Claim 1: "ObservationGroupCfg structure is wrong — should be nested @configclass"

**What the code generates:**
```python
observations = ObservationGroupCfg(
    "joint_pos": ObsTerm(func=mdp.joint_pos),
    "joint_vel": ObsTerm(func=mdp.joint_vel),
)
```
ObservationGroupCfg is instantiated directly with keyword-style obs terms passed as dictionary key–value pairs inside the constructor call.

**What IsaacLab actually requires:**

The official IsaacLab RL tutorial (create_manager_rl_env) shows:

```python
@configclass
class ObservationsCfg:
    """Observation specifications for the MDP."""

    @configclass
    class PolicyCfg(ObsGroup):
        """Observations for policy group."""
        joint_pos_rel = ObsTerm(func=mdp.joint_pos_rel)
        joint_vel_rel = ObsTerm(func=mdp.joint_vel_rel)

        def __post_init__(self) -> None:
            self.enable_corruption = False
            self.concatenate_terms = True

    policy: PolicyCfg = PolicyCfg()
```

The correct pattern is:
- An outer `@configclass` class (`ObservationsCfg`) that inherits from nothing special
- Inner nested `@configclass` classes (one per observation group, e.g. `PolicyCfg`) inheriting from `ObsGroup`
- `ObsTerm` fields are **class attributes**, not constructor kwargs
- The `observations` field on the env cfg is typed as `ObservationsCfg`, not `ObservationGroupCfg`

`ObservationGroupCfg` is actually the base class that `ObsGroup` aliased — you subclass it, you don't instantiate it directly with obs terms.

**VERDICT: BUG CONFIRMED.**
The generated code instantiates `ObservationGroupCfg(...)` directly with string keys, which is structurally incorrect. The API requires a nested `@configclass` inheriting from `ObsGroup` with `ObsTerm` fields as class-level attributes.

### Claim 2: "actions = mdp.joint_positions is gibberish"

**What the code generates:**

From `_RL_TASK_TEMPLATES`, the `actions` template value is the string `"joint_positions"`. This is expanded to:
```python
actions = mdp.joint_positions
```

**What IsaacLab actually requires:**

The `actions` field must be a `@configclass` instance, not a reference to a module attribute. The correct pattern is:

```python
@configclass
class ActionsCfg:
    joint_effort = mdp.JointEffortActionCfg(asset_name="robot", joint_names=["slider_to_cart"], scale=100.0)

actions: ActionsCfg = ActionsCfg()
```

Furthermore, `mdp.joint_positions` does not exist in `isaaclab.envs.mdp`. The actual action config classes are things like `mdp.JointPositionActionCfg`, `mdp.JointVelocityActionCfg`, etc. There is no plain attribute `mdp.joint_positions`.

**VERDICT: BUG CONFIRMED.**
`mdp.joint_positions` is not a valid API symbol, and even if it were, assigning a class attribute directly to a module-level object is not how the IsaacLab manager actions field works. The correct approach is a nested `@configclass` with `Cfg` instances as attributes.

### Claim 3: "Rewards must be dataclass, not dict"

**What the code generates:**
```python
rewards = {
    "reach_target": RewTerm(func=mdp.reach_target, weight=1.0),
    "grasp_success": RewTerm(func=mdp.grasp_success, weight=1.0),
}
```
This is a plain Python `dict`.

**What IsaacLab actually requires:**

The official tutorial shows:
```python
@configclass
class RewardsCfg:
    """Reward terms for the MDP."""
    alive = RewTerm(func=mdp.is_alive, weight=1.0)
    terminating = RewTerm(func=mdp.is_terminated, weight=-2.0)
    pole_pos = RewTerm(
        func=mdp.joint_pos_target_l2,
        weight=-1.0,
        params={"asset_cfg": SceneEntityCfg("robot", joint_names=["cart_to_pole"]), "target": 0.0},
    )
```

The `RewardManager` in IsaacLab reads `RewTerm` objects from class attributes of a `@configclass`, not from a dictionary. Passing a `dict` will cause the manager to fail to detect any reward terms.

**VERDICT: BUG CONFIRMED.**
Rewards must be a `@configclass` with `RewTerm` instances as class-level attributes. A `dict` is structurally wrong for the `RewardManager` protocol.

### Claim 4: "@configclass decorator is missing"

**What the code generates:**

```python
class {task}EnvCfg(ManagerBasedRLEnvCfg):
    ...
```
No `@configclass` decorator is applied to the generated class.

**What IsaacLab actually requires:**

Every configuration class in IsaacLab — including the top-level env cfg — must be decorated with `@configclass`. The tutorial is explicit:
```python
@configclass
class CartpoleEnvCfg(ManagerBasedRLEnvCfg):
    ...
```
The `@configclass` decorator (from `omni.isaac.lab.utils` / `isaaclab.utils`) performs schema validation, default value injection, and enables the `__post_init__` protocol used throughout. Without it, `ManagerBasedRLEnv` will raise errors or silently fail to read configuration fields.

Additionally, all nested configuration sub-classes (`ObservationsCfg`, `ActionsCfg`, `RewardsCfg`, `EventCfg`, `TerminationsCfg`) also require `@configclass`.

**VERDICT: BUG CONFIRMED.**
`@configclass` is entirely absent from the generated code. It is not imported and not applied to the env cfg class or any sub-config classes.

### Claim 5: "No gym.register() generated"

**What the code generates:**

`_generate_isaaclab_env_code` produces only a `{task}EnvCfg` class. No `gymnasium.register()` call is generated anywhere. The `_gen_launch_training` function also produces no registration code — it goes straight to the training subprocess call.

**What IsaacLab actually requires:**

To use `--task Isaac-MyTask-v0` with any training script, the environment must be registered in a task package's `__init__.py` using:

```python
import gymnasium as gym

gym.register(
    id="Isaac-MyTask-v0",
    entry_point="isaaclab.envs:ManagerBasedRLEnv",
    disable_env_checker=True,
    kwargs={
        "env_cfg_entry_point": f"{__name__}.my_env_cfg:MyTaskEnvCfg",
        "rsl_rl_cfg_entry_point": f"{agents.__name__}.rsl_rl_ppo_cfg:MyTaskPPORunnerCfg",
    },
)
```

Without this registration, the training script will raise `gymnasium.error.NameNotFound` when it tries to look up the task name.

**Assessment of severity:** This is a "scaffold completeness" issue as much as a bug — the code generator produces a config class but not the surrounding task package structure needed to actually use it. Whether this counts as a bug depends on intent. If the generated code is presented as a complete, runnable scaffold, it is missing a critical piece.

**VERDICT: BUG CONFIRMED (scaffold incomplete).**
The generated env cfg is not self-sufficient. A `gymnasium.register()` call (and the corresponding `__init__.py` task package) must also be generated for the training launch code to work.

### Claim 6: "mdp.joint_pos is not a real API name"

**What the code generates:**

Observation terms are generated from template obs lists like `["joint_pos", "joint_vel", "ee_pos", ...]`, producing:
```python
"joint_pos": ObsTerm(func=mdp.joint_pos),
"joint_vel": ObsTerm(func=mdp.joint_vel),
```

**What IsaacLab actually provides:**

The official `isaaclab.envs.mdp` API documentation explicitly lists `joint_pos()` as a real observation function:
> `joint_pos()` — "The joint positions of the asset"

The function exists in `isaaclab.envs.mdp` (specifically `isaaclab.envs.mdp.observations`). There is also `joint_pos_rel()` (relative to defaults), but `joint_pos` itself is valid.

Other obs names used in the templates such as `base_lin_vel`, `base_ang_vel`, `projected_gravity`, `joint_vel` are also confirmed real API functions in the same module.

Note: some obs names in the template (`ee_pos`, `ee_ori`, `target_pos`, `target_rel`, `lidar_scan`, `base_pos`) may not map directly to single mdp functions — they are task-specific and would need custom implementations or composition. But the core names like `joint_pos`, `joint_vel`, `base_lin_vel` are real.

**VERDICT: NOT A BUG.**
`mdp.joint_pos` is a real, documented IsaacLab API function. This claim by the previous reviewers is incorrect.

### Claim 7: "isaaclab.train module doesn't exist"

**What the code generates (in `_gen_launch_training`):**

```python
cmd = [
    sys.executable, '-m',
    'isaaclab.train',
    '--task', task,
    '--num_envs', str(num_envs),
    '--max_iterations', str(max_iterations),
    '--log_dir', log_dir,
]
```
This invokes `python -m isaaclab.train`, treating `isaaclab.train` as a Python module.

**What IsaacLab actually provides:**

IsaacLab does not have a `python -m isaaclab.train` entry point. Training is done through framework-specific scripts located at:
- `scripts/reinforcement_learning/rsl_rl/train.py`
- `scripts/reinforcement_learning/skrl/train.py`
- `scripts/reinforcement_learning/sb3/train.py`
- `scripts/reinforcement_learning/rl_games/train.py`

These are invoked via the IsaacLab launcher:
```bash
./isaaclab.sh -p scripts/reinforcement_learning/rsl_rl/train.py --task Isaac-Reach-Franka-v0 --headless
```

There is an `isaaclab_rl` package (as opposed to `isaaclab.train`) that provides environment wrappers for each framework, but it is not a runnable `__main__` module.

Furthermore, `--log_dir` is not a standard argument for these training scripts. Framework-specific arguments differ: RSL-RL uses `--log_root`, SB3 uses its own conventions, etc.

**VERDICT: BUG CONFIRMED.**
`python -m isaaclab.train` does not exist. The generated training launch code would immediately fail with `No module named isaaclab.train`. The correct approach is to invoke the framework-specific train script through the `isaaclab.sh` launcher or a direct `python` call to the script file path.

## Root Cause Summary

The `_generate_isaaclab_env_code` function generates a syntactically plausible but semantically broken IsaacLab env config. The key structural mistakes are:

1. The entire config class hierarchy is missing `@configclass` decorators — without these, IsaacLab's config system cannot validate or instantiate the config.
2. `ObservationGroupCfg` is instantiated directly instead of being subclassed with `@configclass` and nested inside an outer cfg class.
3. The `actions` field is assigned a non-existent `mdp` attribute (`mdp.joint_positions`) instead of a `@configclass` with `ActionCfg` instances.
4. Reward terms are wrapped in a Python `dict` instead of a `@configclass` where terms are class attributes.
5. `_gen_launch_training` invokes a non-existent Python module (`isaaclab.train`) and passes non-standard CLI flags.

The generated code appears to be written by someone who understood the conceptual structure of IsaacLab environments but was not familiar with the actual API conventions (particularly the `@configclass` pattern that pervades the entire manager-based RL framework).
