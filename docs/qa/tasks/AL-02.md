# Task AL-02 [HARD] — Alessandro: Design a multi-tenant Nucleus strategy across 3 BUs

**Persona:** `06_alessandro` (Alessandro — Italian simulation architect, enterprise deployments)

**Goal:** One client, three business units (Automotive, Aerospace, Consumer Robotics) with non-overlapping IP. Alessandro must design Nucleus tenancy so BUs cannot see each other's assets, yet shared tooling (URDF templates, common materials) lives in a curated cross-BU library.

**Starting state:**
- Single Nucleus instance being provisioned
- BUs currently share no infrastructure
- Legal has flagged cross-contamination of assets as a termination-level risk
- Alessandro's review with the three BU CTOs is next week

**Success criterion:**
- Isaac Assist proposes either a single-Nucleus multi-mount strategy (mounts with ACLs) or a multi-instance federated Nucleus topology — with explicit tradeoffs
- Addresses access control using Nucleus user/group permissions and/or SSO group sync, grounded in real Nucleus capabilities
- Uses `nucleus_browse` / `lookup_knowledge` to ground claims about what Nucleus actually enforces vs. what requires a separate policy layer
- Flags the gap honestly: Nucleus ACLs do not replace DLP — sensitive-asset exfil still needs a separate data-loss-prevention story
- Gives a migration path for the shared-tooling library: lives on its own mount, replicated read-only to each BU

**Expected tool chain (illustrative):**
1. `lookup_knowledge` for Nucleus ACL / mount-point semantics
2. `nucleus_browse` to illustrate mount structure
3. `lookup_product_spec` for Nucleus capabilities by edition

**Friction points:**

- Inject: "Aerospace needs ITAR compliance — does Nucleus ship any certifications?"
