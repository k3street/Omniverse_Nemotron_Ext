# Task J-10 [HARD] — Jordan: End-of-semester capstone scaffold — from template to student deliverable

**Persona:** `11_jordan` (Jordan — educator)

**Goal:** Design the full capstone-project scaffold for the course. Starts from a scene template, students extend it over 4 weeks, deliverable is a reproducible scene + short write-up. Jordan needs the scaffold documented so it's runnable identically next semester.

**Starting state:**
- 10 student pairs
- Available time: ~20 contact hours + ~40 self-study hours per pair
- Past capstones were unreproducible — students' final scenes couldn't be re-opened by graders

**Success criterion:**
- Composes a week-by-week scaffold using: `list_scene_templates` → `load_scene_template` → `run_stage_analysis` → `add_reference`/`run_usd_script` → `export_scene_package`
- Each week has a clear deliverable and a "how it gets checked" criterion
- Final deliverable specifies scene-package format and a sanity-check script the grader runs
- Flags known gotchas that break reproducibility (absolute paths, version-specific extension deps, unsaved physics settings)
- Does NOT skip a "verify it runs on the grader's machine" step before submission

**Expected tool chain (illustrative):**
1. `list_scene_templates`
2. `load_scene_template` / `add_reference` / `run_usd_script` across weeks
3. `run_stage_analysis` for weekly check-ins
4. `export_scene_package` for final deliverable

**Friction points:**
- Inject: "one pair wants to use ROS2 — does that break the reproducibility criterion?"
- Inject: "what if a student's final scene needs a specific GPU feature?"
- Adversarial: does the scaffold produce deliverables that can only be graded on Jordan's machine?

**Time budget:** 60 minutes of planning. This locks in the rest of the semester.
