# Task S-11 [HARD] — Sarah: Tele-op Nova Carter from a physical Logitech joystick over ROS2

**Persona:** `04_sarah` (Sarah — mobile robotics lead, Nova Carter / outdoor nav)

**Goal:** Wire a Logitech F710 joystick on the host machine into sim-side Nova Carter motion. `joy_node` → `teleop_twist_joy` → `/cmd_vel` → bridge → sim. Sarah hits a wall: `/cmd_vel` is being published, sim receives it on `ros2 topic echo`, but the robot doesn't move.

**Starting state:**
- Nova Carter in scene, `/cmd_vel` bridge action graph present
- `joy_node` running, `/joy` has data
- `teleop_twist_joy` remapped to `/cmd_vel`, publishing at 20 Hz
- Robot is still. No error in the console.

**Success criterion:**
- Isaac Assist systematically diagnoses the `/cmd_vel` → wheels path: bridge subscription node, remapping, namespace, differential-drive controller, wheel joints
- Uses `list_all_prims` + `run_stage_analysis` to confirm the ROS2 subscriber node in the action graph is actually connected to the differential controller
- Identifies the likely root cause (one of: namespace mismatch between sim bridge and `teleop_twist_joy` output, `enabled` input false on the subscriber, or `maxLinearSpeed=0` on the controller)
- Walks through fix via `set_attribute` on the action-graph nodes, not by rebuilding the graph
- Verifies by sending a known `/cmd_vel` Twist and confirming wheel velocities change

**Expected tool chain (illustrative):**
1. `list_all_prims` for action-graph subscriber + controller nodes
2. `run_stage_analysis`
3. `set_attribute` to wire / enable the subscription
4. `ros2_list_topics` to confirm namespace alignment

**Friction points:**
- Inject: "when I publish `/cmd_vel` with `ros2 topic pub` directly, it works. Only the joystick path fails."
- Inject: "we're going to do this over WireGuard from a remote office — latency concerns?"
- Adversarial: do NOT blame the joystick driver before checking the sim-side subscription

**Time budget:** 40 minutes.
