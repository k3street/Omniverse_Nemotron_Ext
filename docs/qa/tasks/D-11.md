# Task D-11 [MEDIUM] — Dimitri: Look up IMU datasheet and spec the sim equivalent

**Persona:** `12_dimitri` (Dimitri — lidar/sensor specialist)

**Goal:** Client AMR uses a Bosch BMI088 IMU. Dimitri wants the sim IMU prim configured with matching noise density, bias instability, and output rate so the robot_localization EKF behaves the same in sim as on hardware.

**Starting state:**
- IMU mounted at `/World/amr/base_link/imu_link`, currently using default Isaac Sim noise (approximately none)
- BMI088 spec sheet values:
  - Gyro: noise density 0.014 °/s/√Hz, bias stability ≈ 2 °/hr, output ≤ 2 kHz
  - Accel: noise density 175 µg/√Hz, bias stability ≈ 20 µg, output ≤ 1.6 kHz
- EKF subscribes at 200 Hz

**Success criterion:**
- Uses `lookup_product_spec` for BMI088
- Configures the IMU prim with the listed noise density and bias stability (or names the Isaac Sim fields that take these values)
- Sets output rate to 200 Hz (matches EKF input; higher is wasteful)
- Specifies frame_id `imu_link` for ROS2 publishing
- Flags limits: Isaac Sim's IMU noise model is Gaussian white + random walk — does not capture BMI088's temperature drift
- Does NOT leave bias instability at 0 (would make sim EKF overconfident vs real)

**Expected tool chain (illustrative):**
1. `lookup_product_spec` for BMI088
2. `add_sensor_to_prim` or `configure_camera`-equivalent for IMU
3. `ros2_publish` to expose `/imu/data` at 200 Hz

**Friction points:**
- Inject: "can I add a temperature-drift term manually via a Python callback?"
- Inject: "is 200 Hz enough or should I run 400 Hz and downsample?"
- Adversarial: does it report noise values in the wrong units (°/s vs rad/s, mg vs m/s²)?

**Time budget:** 25 minutes.
