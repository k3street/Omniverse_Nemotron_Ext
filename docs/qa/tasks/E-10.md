**Persona:** `02_erik` (Erik — startup CTO, humanoid warehouse picking)

**Goal:** The customer walkthrough runs on a loop on a big screen in the meeting room. Erik needs a script that plays a pick-place cycle, resets, and plays again — for 30 minutes straight without stalling, visual glitches, or crashing.

**Starting state:**
- Demo scene working (from E-08 packaging)
- Demo laptop: RTX 5090, plenty of headroom
- Current script runs ONE cycle and exits
- No fault handling

**Success criterion:**
- Provides a loop wrapper that: (1) resets scene state each cycle, (2) catches exceptions and auto-restarts the cycle instead of crashing, (3) throttles to real-time so it doesn't run 10× too fast for a human audience
- Uses `set_joint_targets` / `set_attribute` for scripted motion — not an RL policy (the demo is scripted, not trained)
- Mentions `omni.kit.app.get_app().get_update_event_stream()` or similar for time-sync'd stepping
- Warns about memory leaks over 30 minutes — Replicator / texture caching / random variants can grow; recommends a process-level restart every N cycles as a belt-and-braces fallback

**Expected tool chain (illustrative):**
1. `generate_code_snippet` for the cycle loop
2. `set_joint_targets` / `set_attribute` for waypoints
3. `check_physics_health` after a 5-min dry run
4. `lookup_knowledge` for Isaac Sim app-update event stream

**Friction points (probe these mid-conversation):**
- Inject: "can we skip reset and just let the scene drift?" — no, after 10 cycles the boxes are everywhere; reset is non-negotiable
- Adversarial: "just use `while True: time.sleep(0.1)`" — explain why that desynchronizes from the renderer
- Demo reality: "what if the laptop goes to sleep?" — caffeine / `systemd-inhibit` / disable power management on the demo machine
