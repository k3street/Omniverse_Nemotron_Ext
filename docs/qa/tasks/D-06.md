# Task D-06 [HARD] — Dimitri: Generate a 2D occupancy map for Nav2 from the warehouse scene

**Persona:** `12_dimitri` (Dimitri — lidar/sensor specialist)

**Goal:** Produce a `map.pgm` + `map.yaml` occupancy grid from the warehouse USD, suitable for loading into Nav2's `map_server`. Resolution 0.05 m/px, origin aligned with scene origin.

**Starting state:**
- Warehouse scene: ~40 m x 25 m, walls, shelves, static obstacles
- Target AMR footprint: circular, r=0.25 m — so inflation will be handled downstream
- Dimitri wants a one-shot generation, not "drive the robot around and SLAM it"

**Success criterion:**
- Uses `generate_occupancy_map` with resolution 0.05, bounds matching the scene
- Output: PGM (grayscale, 0/205/254 for occupied/unknown/free per Nav2 convention) + YAML with `resolution`, `origin`, `free_thresh`, `occupied_thresh`
- Origin in YAML matches the scene's world origin (or explicitly offset with justification)
- Flags that dynamic prims (moving forklifts) should be excluded from the map
- Does NOT require launching a ROS2 SLAM stack

**Expected tool chain (illustrative):**
1. `run_stage_analysis` to identify static vs dynamic prims
2. `generate_occupancy_map` with static-only filter
3. Output validation (PGM dimensions match scene bounds / resolution)

**Friction points:**
- Inject: "the shelves have gaps the lidar sees through at 0.3 m height — is the slice height configurable?"
- Inject: "can I export at 0.025 m/px for tighter planning?"
- Adversarial: does it produce a map with origin at wrong sign convention (Nav2 uses lower-left, not center)?

**Time budget:** 40 minutes.
