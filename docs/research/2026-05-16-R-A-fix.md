## ┬¦4 Recommendation ŌĆö lint extension for Pydantic-validation per tool call

**Problem this audit found**: The existing lint gate catches structural JSON issues (missing
required top-level fields, type mismatches at the template level) but does NOT parse the
Python-like pseudocode in `code`/`code_template` to validate tool-call kwargs against
`_models.py` Pydantic schemas. This caused 10 schema mismatches across 6 templates to ship
undetected.

**Design sketch** for a `--validate-tool-calls` lint pass:

1. **Extract tool calls from `code` field**: Use a simple AST parse (`ast.parse`) or
   regex to find all `tool_name(...)` call expressions in the `code` string.

2. **Map to Pydantic model**: Look up each `tool_name` in the `TOOL_ARGS_MAP` dict already
   present at the bottom of `_models.py` (line ~3984). This gives the `Args` class.

3. **Validate kwargs**: For each call, extract keyword arguments. Check:
   - All required fields (non-Optional, no default) are present.
   - No field named in the call that is missing from the schema (even if `extra='allow'`
     absorbs it, flag as WARN so authors know the kwarg is a no-op).
   - `axis` field: if schema type is `List[float]`, reject string values like `"Z"`.

4. **Template substitution caveat**: `code_template` contains `{{role.field}}` placeholders
   that are not valid Python. The lint pass should skip `code_template` or substitute
   placeholder values with schema-compatible dummy values before AST parsing.

5. **Integration**: Add as a secondary pass in `lint_canonical_templates.py` under
   `--strict` flag (or always-on for new templates). Output BLOCKER for missing required
   fields, WARN for unrecognised extra kwargs.

Estimated scope: ~150 lines of new lint code. Would have caught all 10 issues in this
batch automatically.
