## §3 Scope decisions (larger changes)

### P7 — A8 bounding_box_height pivot (~35 LOC)

Choice: **option (b)** — reduce scope, not add a new target_source.

Rationale: adding `bounding_box_height` to the controller handler requires editing
`tool_executor.py`, adding a new code path, testing under Kit RPC, and potentially
extending the enum in `tool_schemas.py`. That's function-gate work, not form-gate.
Option (b) delivers the same canonical intent (height-ranked destacking) using two
already-verified tools (`get_bounding_box` + `curobo`). The ordering is
pre-computed before controller install, so `source_paths` arrives already ranked.

Side effect: `transit_height`, `top_down_approach`, `grasp_z_offset` are no longer
explicit kwargs; cuRobo handles approach geometry internally. The `settle_ticks`
concept (wait for pile to settle) is not wired — add a `sim_control` step-and-settle
block in a future patch if needed.

### P8 — A6 handoff gating stripped

Choice: remove unsupported kwargs rather than stub them.

Rationale: `handoff_signal`, `claim_mutex`, `activation_signal` are not in
`setup_pick_place_controller` schema, and the handler does not read them. Keeping
them creates intent illusion: the template looks like it enforces serialisation but
doesn't. The `setup_robot_claim_mutex` / `setup_robot_handoff_signal` calls remain —
they install the USD-level gating markers which a future handler extension or
orchestrator script can activate. This is honest about the current limitation.

## §4 What audit #3 may still find

1. **A8 code_template** uses `get_bounding_box(prim_path=…)` — verify schema. If
   `get_bounding_box` requires a different kwarg name (e.g. `path`), this needs
   another round. (Audit #2 didn't cover this call site.)

2. **A6 `setup_robot_claim_mutex` / `setup_robot_handoff_signal` kwargs** — audit
   #1/2 verified these were fixed in R-A-fix, but the P8 code_template block
   uses `resource_path=` for claim_mutex. If the schema changed since the R-A-fix
   patch, there may be a residual mismatch.

3. **A5 `emit_ros2_control_yaml` kwargs** (`joint_states_topic`, `joint_commands_topic`,
   `update_rate_hz`) — not checked in audit #1 or #2. These may or may not be in
   schema.

4. **A7 `iterate_reward` kwargs** (`user_feedback`, `prev_reward_code`, `metrics`) —
   audit #2 did not cross-check iterate_reward schema exhaustively; only generate/evaluate
   were flagged in audit #1.

5. **A8 `create_heap_zone` kwargs** — `radius`, `n_items`, `item_size` were not
   schema-checked in either audit round. If schema uses different names the heap
   won't instantiate.

6. **A1 `drop_targets` shape** — dict keyed by cube_path → [x,y,z]. Schema says
   "dict mapping cube_path → world drop position". Valid per description, but if
   handler expects a list instead, all 6 drop targets will be ignored.
