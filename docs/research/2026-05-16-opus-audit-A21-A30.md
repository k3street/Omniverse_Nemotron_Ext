## §1 Methodology

1. **Spot-read 4 deeply** — A21 (teach-pendant), A22 (zone-partition, already
   post-patched), A28 (welder seam-track), A30 (assembly-constraint-verify).
   Read every field; cross-checked tool args against the live schema in
   `service/isaac_assist_service/chat/tools/tool_schemas.py` and pydantic
   models in `…/handlers/_models.py`.
2. **Sample 6 with one angle each** — A23 (Angle B prim-path), A24 (Angle E
   cross-template), A25 (Angle D honesty), A26 (Angle F semantic realism),
   A27 (Angle C code-template substitution), A29 (Angle A tool-call
   correctness).
3. **Form-gate validation** — Re-ran the lint script on all 10 to confirm the
   claimed clean pass and to bound what the gate can/cannot see.
4. **Cross-checked handler implementations** for tools whose schema and
   pydantic models disagree (e.g. `interpolate_trajectory.waypoints[]`,
   `validate_assembly_constraint.type` enum, `teleop_safety_config.workspace_limits`).

The form-gate (`lint_canonical_templates.py:202-253`) inspects only:
- presence of required pydantic fields (`model.model_fields[*].is_required()`),
- absence of unknown kwargs (`kwargs - known_fields`).

It does NOT inspect enum values, nested-object schemas, or per-list-item
required keys. Most of the findings below sit in that blind spot.

## §2 Findings per Angle

### Angle A — Beyond-schema tool-call correctness

**A21 / interpolate_trajectory — BLOCKER**
`workspace/templates/CP-NEW-teaching-mode-pose-replay.json:19, 78`
Code passes `waypoints=[{"pose_name": "home"}, …]` to
`interpolate_trajectory`. The schema
(`tool_schemas.py:4647-4675`) declares
`waypoints[].joint_positions` as **required** for each item; the codegen
handler `robot.py:3086` dereferences `wp["joint_positions"]` directly
and will `KeyError` on the first iteration. Pydantic accepts the call
(`InterpolateTrajectoryArgs.waypoints: List[Dict[str, Any]]`,
`extra='allow'`) and the form-gate has no per-item-required check, so the
lint passed. The template's own `failure_modes[1]` flags this risk in
prose but the code still ships the broken form. Effect: A21 cannot execute
past `interpolate_trajectory`. **Severity: BLOCKER.**

**A22 / teleop_safety_config — BLOCKER**
`workspace/templates/CP-NEW-zone-partition-safety.json:19, 20`
Code passes
`workspace_limits={"x_min": -0.9, "x_max": 0.9, …}`. The schema
(`tool_schemas.py:2858-2877`) declares the field as an object with
properties `{min: [x,y,z], max: [x,y,z]}` — i.e. a flat AABB with two
length-3 arrays. The keys `x_min/x_max/y_min/y_max/z_min/z_max` do not
exist in the schema; the handler will either reject or silently ignore
them. Both the `code` and `code_template` fields carry the wrong shape.
**Severity: BLOCKER.**

**A30 / validate_assembly_constraint — BLOCKER**
`workspace/templates/CP-NEW-assembly-constraint-verify.json:17, 18`
Code calls `validate_assembly_constraint(type="cylindrical_fit", …)`.
The schema (`tool_schemas.py:9705-9731`) restricts `type` to an enum:
`["coincident_axes", "concentric", "tangent", "parallel_planes",
"fixed_offset", "angle_between", "distance_between"]`.
`cylindrical_fit` is NOT a member — pydantic will raise on dispatch and
the lint missed it because it doesn't read enums. The natural match for
peg-in-bushing is `concentric` (axis + tolerance) or `coincident_axes`.
The template even imagines `revolute_fit` / `prismatic_fit` in its own
failure_modes — both also invalid. **Severity: BLOCKER.**

**A28 / vision_plan_trajectory output unused — MED**
`workspace/templates/CP-NEW-welder-seam-track.json:32` (Phase 4 loop)
Calls `vision_plan_trajectory` three times in a `for seg_idx in range(3)`
loop but discards the return value (no waypoint stitching, no
correction applied to the subsequent `follow_trajectory_with_compliance`
trajectory). The "adaptive correction" the goal advertises is not in
the code. Semantic-only failure; pure tool-call form is fine.
**Severity: MED (goal–code drift).**

### Angle B — Prim-path consistency

**A22 verify_args has misleading pick/place paths — LOW**
`…/CP-NEW-zone-partition-safety.json:36-42`
`verify_args.stages[0].pick_path = "/World/ZoneBoundarySensor"`,
`place_path = "/World/HumanProxy"`. This canonical has no pick or
place; it is a safety-stop demo. The fields appear to be wedged in to
satisfy a verifier that expects them. The extra keys
`zone_sensor_path`, `expected_stop_on_entry`, `clearance_mm` are the
honest ones. Leaving the stages[] field in place creates a misleading
read of the canonical's intent. **Severity: LOW.**

**A30 verify_args uses null robot_path/robot_kind — LOW**
`…/CP-NEW-assembly-constraint-verify.json:31-36`
`robot_path: null, robot_kind: null` — semantically honest (no robot),
but if downstream verifiers `.lower()` the robot_kind they'll crash.
Worth a defensive empty string. **Severity: LOW.**

### Angle C — Code-template substitution coherence

**A27 code_template is a stub placeholder — HIGH**
`workspace/templates/CP-NEW-palletizer-mixed-sku.json:code_template`
The `code_template` field is the single line
`"# template placeholder — see code field above for fully-worked implementation"`.
The hard-instantiate / typed-variable resolver path expects to render
`code_template` with `{{role.path}}` substitutions; with a stub, the
template cannot be instantiated under a non-default role binding. The
`code` field is 5731 chars and complete; the gap is purely in the
template-form. All nine sibling A21–A30 templates supply a real
`code_template`. **Severity: HIGH (cross-template inconsistency +
functional gap for instantiation).**

**A24 `code_template` quoting drift — LOW**
`workspace/templates/CP-NEW-gravity-dispenser-feeder.json:161`
Template renders `robot_name={{primary_robot.class}}` with no quotes
around the string-typed render target (compare with sibling A21:
`robot_name="franka_panda"` literal). When the placeholder is replaced
with raw text, `robot_name=franka` will be a NameError. Several
positional fields in the template have the same drift
(`dest_path={{…}}`, `sensor_path={{…}}`, etc). Renderer may quote, but
if not, every call is broken. **Severity: LOW (renderer-dependent).**
Same pattern (no surrounding quotes for string fields) recurs in
**A23**, **A25**, **A29** code_templates — common across the batch but
likely intentional convention; flag for verifier confirmation.

### Angle E — Cross-template consistency / naming drift

Uses `"motion_controllers": {"curobo": "untested"}`.

**A25 verify_args/diagnose_args are literal `null` — MED**
`…:43, 55` Both fields are JSON `null`. Form-gate's `C1_EMPTY_CORE_FIELD`

### Angle F — Semantic realism / would-it-run

**A21 — Conceptually broken even if interpolate were fixed.**
`teach_robot_pose` snapshots the *current* joint configuration of the
robot at call time (`tool_schemas.py:3860-3870`). A21 calls it six
times in immediate succession without ever moving the robot between
calls — so all six saved poses will be the same configuration. The
"home → above_source → pick → lift → place → drop" sequence is a
labeling exercise only; no real teach happens. The replay will trace
a six-step single-point trajectory.
Real fix: between each `teach_robot_pose`, the canonical needs to
`set_joint_targets(…)` or `move_to_pose(…)` to actually drive the EE
to the labeled location, then settle, then teach. **Severity: HIGH —
semantic-blocker.**

**A22 — Already patched once for `set_linear_velocity`; remaining gap.**
`teleop_safety_config` is documented as "only active during a teleop
session" (own failure_modes). The canonical never calls
`start_teleop_session`, so the watchdog has no session to police. The
zone trigger fires the sensor flag but no controller reads it. Demo
shows "the inhibit fires" — in reality nothing reduces Franka velocity.
**Severity: MED (canonical is a sensor/visualization demo, not the
safety-stop the goal claims).**

**A24 thoughts↔code drift on `target_xy` — LOW.**
Thoughts say "target_xy=[0.0, 0.0]" (drop on belt); code passes
`target_xy=[0.0, -0.55]`. Belt y-range is centered at 0; dispenser drop
at y=-0.55 may be past the belt edge (belt size [0.35, 1.5, 0.05] →
y-bounds [-0.75, +0.75]; still inside). Dispenser drops upstream of
the sensor at y=+0.65; belt velocity [0, 0.15, 0] carries parts toward
sensor in ~8s each. The mismatch is the design doc, not the executable
code. **Severity: LOW.**

**A24 target_source contradiction — LOW.**
Goal narrative: "belt-gated setup_pick_place_controller (sensor_gated
mode)". Code passes `target_source="curobo"` + sensor_path. The schema
allows sensor_path under "native/sensor_gated mode"
(`tool_schemas.py:3791`); using it with curobo is unusual but not
strictly disallowed. Behavior depends on handler. Confirm at runtime.
**Severity: LOW.**

**A29 — Mesh tools on Cube prims.**
A29's own failure_modes admit `fix_collision_mesh` and
`compute_convex_hull` may fail on a USD Cube (not a `UsdGeom.Mesh`).
The canonical builds all three test prims as Cube. Best-case the tools
no-op silently; worst-case they error. The canonical is honest about
this, but the function-gate verification (`expected_fatal_issues=0`)
sets up the verifier to either rubber-stamp or false-fail. **Severity:
MED.**

## §5 Verdict

**RED — 3 BLOCKERs in 10 templates is a higher rate than A9–A20**
(1 BLOCKER in 12). The schema form-gate is doing its narrow job
(`is_required` + unknown kwargs only) but is structurally blind to:

- enum-value violations on string fields,
- per-list-item required keys (nested-array schemas),
- object-shape validation for nested-object schemas,
- semantic correctness of tool call sequences (Angle F class).

A21, A22, A30 will fail at runtime on the first contact with the actual
handler dispatch. Until those three patches land, A21/A22/A30 should
not enter the canonical library counted as "ready to test".

A25 (motion_controllers shape) and A27 (stub code_template) are
HIGH-severity structural breaks that block downstream tooling (verifier
filters, hard-instantiate path).

The remaining 5 (A23, A24, A26, A28, A29) are runnable in principle
but carry MED/LOW semantic notes worth fixing before function-gate.

## §6 Form-gate effectiveness — did it catch what it claimed?

Form-gate as run (`scripts/lint_canonical_templates.py --validate-tool-calls`)
reports `10 OK, 0 ERROR, 0 WARN, 0 INFO`. The claim of "pre-draft
schema lookup" appears to have been honored — the templates do call
real tool names with real required-field kwargs and no unknown kwargs.
That part is real and successful.

**What the form-gate cannot see, and missed:**

1. **enum values** — `validate_assembly_constraint.type="cylindrical_fit"`
   (A30). The schema's `enum: [...]` is ignored by
   `lint_canonical_templates.py:225-253`, which validates only against
   the pydantic model's `model_fields` set.
2. **nested-object property names** — `teleop_safety_config.workspace_limits`
   uses keys `x_min/x_max/y_min/y_max/z_min/z_max` (A22) instead of the
   schema's `min/max` arrays. The pydantic model declares the field as
   `Dict[str, Any]` so all keys pass.
3. **nested-array per-item required fields** —
   `interpolate_trajectory.waypoints[].joint_positions` (A21).
   The pydantic model declares `waypoints: List[Dict[str, Any]]` and
   `extra='allow'`, so the per-item required field is invisible to it,
   and the lint script doesn't probe inside list items at all.
4. **semantic / control-flow correctness** — A21 teach-loop without
   intervening joint moves, A22 watchdog without teleop session, A28
   vision output unused. These are out of scope for any AST gate.
5. **cross-template structural fields** — A25's `motion_controllers`
   shape, A27's stub `code_template`, divergent `verified_status`
   phrasing. The lint inspects core-six presence but not motion-controller
   substructure or template-rendering fitness.

So: **form-gate did exactly what its code does; the claim "schema-level
should be clean" is true; the assumption that "schema-clean ≈ correct"
underestimated the gap.** Recommend lifting these checks into the lint
script before the next batch:

- enum-value check against `tool_schemas` JSONSchema enum lists.
- nested-object key check against JSONSchema `properties`.
- nested-array per-item required check against JSONSchema
  `items.properties` + `items.required`.
- structural lint on `motion_controllers` shape.
- non-empty `code_template` content check (not just presence).
