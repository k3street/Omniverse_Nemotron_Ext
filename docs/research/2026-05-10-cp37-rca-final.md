# CP-37 plan_pose Fails 24/24 — Root Cause

**Date:** 2026-05-10
**Method:** Isolated plan_pose tests with synthetic poses + real CP-37 scene.

## Conclusion

**It's a reach problem, not an obstacle problem.**

CP-37 spawns 4 cubes at world x ∈ {-1.4, -1.15, -0.9, -0.65}, y=0.4, z=0.835.
Franka robot at world [0, 0, 0.75], rotated 90° around Z.

In Franka's BASE frame (after applying inverse rotation):
- Cube_1: base = [0.4, **1.4**, 0.085]  → distance 1.456 m
- Franka max reach: 0.855 m
- **Distance exceeds reach by 0.6 m**

cuRobo plan_pose fails not because of scene-floor obstacles, but because the
goal pose is **physically unreachable** for the Franka manipulator from its
fixed base position.

## Validation Tests

| Test | Goal world | Goal base | Empty scene | Just Pillar | Full scene |
|---|---|---|---|---|---|
| Cube_1 | [-1.4, 0.4, 0.835] | [0.4, 1.4, 0.085] | ❌ | ❌ | ❌ |
| Bin | [0, -0.4, 0.825] | [-0.4, 0, 0.075] | ✅ | ✅ | ✅ |
| Easy point (0.4 fwd) | [0.4, 0, 0.85] | [0, -0.4, 0.1] | ✅ | ✅ | ✅ |

When goal is **within reach**, plan_pose succeeds **with full obstacle scene**.
When goal is **out of reach**, plan_pose fails **even with empty scene**.

## Why Last Night's Sensor Memory Was Misleading

Session-end memory said "cuRobo plan_pose fails 24/24 in real-handler context
with scene-floor obstacles". The "scene-floor obstacles" theory was wrong —
isolated test proves scene_cfg has no effect. The 24/24 failure is reach-bound.

The earlier finding "isolated test (just Pillar registered, home q) → 4/4 succeeds"
was probably testing a DIFFERENT goal pose (not cube positions far from robot).

## Implications for Phase 4

The `obstacle_rich` profile (skip Table/Belt/Bin) does NOT help CP-37 because
the issue is reach, not obstacles.

The `high_speed_belt` profile needs more than lookahead-x. It needs:
1. **Predictive planning**: when cube is approaching reach radius but not yet
   inside, predict its position 0.5s ahead and plan for that.
2. **Phase change**: controller should hold approach_above pose ready and
   execute the descent only when cube enters reach.

This is **not a one-line config**. It's a meaningful change to the controller's
state machine. Phase 4 spec needs to be updated to reflect this.

## Implications for Other Plan_Fail CPs

11 of 24 patched-set CPs had P-PLAN_FAIL pattern in RCA. Likely most are
reach-bound (cubes spawned far away on belts, not yet within reach). These
need the SAME predictive planning fix.
