## Top 3 findings

### F1 (HIGH) — A54: conveyor_track segment prim-path mismatch

**Affected**: `CP-NEW-sbend-sortation.json`

**Evidence**: `robot.py:1764-1839` (`_gen_create_conveyor_track`) creates segments under
`'/World/ConveyorTrack'` as parent Xform with children
`'/World/ConveyorTrack/Segment_0'`, `'/World/ConveyorTrack/Segment_1'`, etc.

The template's body and code_template both call:
```python
setup_zone_partition(
    conveyor_path="/World/ConveyorTrack_Segment_0",  # WRONG
    n_zones=2,
    robots=["lane_a_zone", "lane_b_zone"],
)
```

Handler `_handle_setup_zone_partition` (`robot.py:5618-5677`) does
`stage.GetPrimAtPath('/World/ConveyorTrack_Segment_0')` → invalid → prints
`{"error": "conveyor not found: '/World/ConveyorTrack_Segment_0'"}` and `raise SystemExit`.

Failure_modes entry #1 even warns about this exact pattern but the body uses the wrong
form. This is a copy-paste hazard if any other future template re-uses the same anchor.

**Patch**: Replace with `conveyor_path="/World/ConveyorTrack/Segment_0"` in BOTH `code` and
`code_template`. Update failure_modes #1 to mention the slash-delimited child convention.

### F2 (MEDIUM) — A57: extract_attention_maps stub-PNG fallback masks real failures

**Affected**: `CP-NEW-groot-attention-maps.json`

**Evidence**: When `capture_viewport` doesn't return a base64 image (no live Kit), the
template writes a 72-byte stub PNG (`b"\x89PNG\r\n\x1a\n" + b"\x00" * 64`) to
`/tmp/groot_attn_obs.png`. `extract_attention_maps` handler then loads this with PIL,
which will raise `PIL.UnidentifiedImageError` because IHDR chunk is missing.

The body assertion `len(weights) > 0 and os.path.exists(overlay_path)` is enforced by
the code_template's hard `assert`, but only the body code soft-warns. The body code
falls through to "stub overlay" if not present and reports SUCCESS to stdout even when
weights={}. This is honesty drift — code_template hard-fails, code soft-passes.

Also worth flagging: when running in form-gate mode (no checkpoint), assertion will
ALWAYS fail because no real weights dict comes back. verified_status correctly says
"function-gate pending" but should add "form-gate BLOCKED (extract_attention_maps
requires a real .pt checkpoint to return non-empty weights)".

**Patch**: Make `code_template` assertion soft (warning instead of assert), OR remove
the stub-PNG path and gate cleanly on checkpoint presence. Add HARD BLOCKER for
weights-dict-content to verified_status.

### F3 (MEDIUM) — A52/A53: robot_wizard description claims franka-only but registry supports ur10e

**Affected**: `CP-NEW-ros2-moveit2-franka.json` (informational),
`CP-NEW-ros2control-hw-mock.json` (informational; ur10e DOES work)

**Evidence**: `_models.py:1589` `RobotWizardArgs.robot_name` description says
"Currently supports: 'franka_panda' (aliases: 'franka', 'panda')". But
`_shared.py:188` shows `"ur10e"` in `_ROBOT_WIZARD_REGISTRY` with full asset URLs.
A53 calls `robot_wizard(robot_name="ur10e", ...)` and this WILL succeed because
the handler at `robot.py:823-840` reads the registry directly.

This is a **schema-doc drift** issue, not a template bug. The template author was
correct to use it; the schema description is misleading. Future agents reading the
schema description will be told "only franka_panda" and may wrongly avoid this path.

**Patch**: Update `RobotWizardArgs.robot_name` description in `_models.py` to enumerate
all registry keys (franka_panda, ur10e, nova_carter, …). No template change required.

## Other findings (LOW severity)

### F4 (LOW) — A56: list_bridges return shape mismatch

`bridge_tools.py:264` returns `{bridges: [...], n_alive: N}` — a dict with a `"bridges"`
key, NOT a flat list. Template body does `next((b for b in bridges if ...), None)` where
`bridges = list_bridges()`. Iterating a dict yields the keys ("bridges", "n_alive") —
the predicate `b.get("bridge_id", "")` will throw `AttributeError: 'str' object has no
attribute 'get'`.

The template treats this as informational ("non-fatal — may not appear in list_bridges")
so the code path never asserts on it, but the code DOES try to iterate. Likely
`AttributeError` will be raised before the informational print fires.

**Patch**: `bridges = list_bridges().get("bridges", [])` instead of `bridges = list_bridges()`.
The same issue exists in A60 (`bridges = list_bridges(); next(b for b in bridges if b.get(...))`)
— same fix.

### F5 (LOW) — A60: list_bridges same flat-vs-nested mismatch as F4

Identical fix to F4 in `CP-NEW-mqtt-sparkplug-iot.json`. The template will raise
AttributeError on the `next((b for b in bridges if b.get("bridge_id") == bid), None)`
line, blocking the kind-validation gate from ever running.

### F6 (LOW) — A55: import_robot file_path "UR10" works case-insensitively but the
asset library mapping is to `ur10.usd` (UR10, not UR10e). A55 also asserts
`"singularity_threshold_danger": 100` etc in settle_state — these are documentation
constants, not handler-readable. Acceptable. The robot.path in role_defaults is
`/World/UR10` (consistent with import_robot dest_path); body and template agree.
Cross-template naming drift CHECK: zero drift.
