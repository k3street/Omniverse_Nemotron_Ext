## §2 Findings per Angle

### Angle A — Beyond-schema tool-call correctness

**A33 / diagnose_opcua_bridge — BLOCKER**
`workspace/templates/CP-NEW-machine-tender-door-interlock.json:22, 298`
Template calls `diagnose_opcua_bridge(bridge_id="opc.tcp://localhost:4840")`
i.e. passes the OPC-UA URL as the bridge_id. The handler
(`bridge_tools.py:497-528, 531-534, 121-149`) generates `bridge_id` via
`_make_id()` and writes `<bridge_id>.pid`, `<bridge_id>.log` files. The URL
string `"opc.tcp://localhost:4840"` is never written as a filename — diagnose
will fail with `error: "no such bridge_id"` (matches the modbus diagnose code
path it reuses). The attach call's return value (`{ok, bridge_id, pid, log_path,
url, …}`) is NOT captured anywhere in the template's `code` or `code_template`.
Same issue applies to `bridge_pause/bridge_resume` if used — but A33 uses
`sim_control` for that step, so only diagnose is reachable-and-broken.
**Severity: BLOCKER.** Fix: capture attach result, pass `result["bridge_id"]`
to diagnose.

**A37 / setup_pick_place_controller.color_routing — BLOCKER**
`workspace/templates/CP-NEW-nir-material-divert.json:20`
Template labels items with `set_semantic_label(class_name="metal",
semantic_type="material")` which creates a `Semantics_material` USD instance
(handler `scene_authoring.py:3124`: `instance_name = f"Semantics_{semantic_type}"`).
The cuRobo controller's color-routing dispatch (`pick_place.py:3192-3221`,
`_cube_semantic_class`) only inspects three instance names:
`Semantics_color`, `Semantics_colour`, `Semantics_class`. It does NOT read
`Semantics_material`. Result: every item falls through to `destination_path`
(`/World/BinPlastic`); metal and glass items will never reach their bins.
The whole NIR-routing premise is non-functional. The template's
`failure_modes` ack the semantic_type symmetry but miss that the controller
itself is hard-wired to color/class instance names.
**Severity: BLOCKER.** Fix: use `semantic_type="class"`, or extend
`_cube_semantic_class` to also probe `Semantics_material`.

**A35 / export_dataset semantic mismatch — MED**
`workspace/templates/CP-NEW-groot-export-dataset-hdf5.json:20`
`export_dataset` (`tool_schemas.py:2760`) materialises Replicator RGB+annotator
frames — synthetic data generation output. The template's `goal` and
`task_id` claim an HDF5 robot-demo bundle for Isaac Lab training, then calls
`export_dataset(output_dir=/tmp/groot_export_hdf5, num_frames=50)` to produce
it. These are two different pipelines: GR00T HDF5 demos come from
`record_teleop_demo` writers, not Replicator. The template's own thoughts
admit it ("must be called AFTER create_sdg_pipeline in a real SDG workflow")
but proceed anyway. Pass-condition is just "directory non-empty after call",
so the structural check will pass while delivering wrong content type.
**Severity: MED — semantic realism / honesty about deliverable shape.**

**A39 / interpolate_trajectory + register_moving_obstacle composition — LOW**
`workspace/templates/CP-NEW-moving-conveyor-pick.json:20`
Both calls are schema-clean — the per-item required `joint_positions` is
present (post-A30 lint catches this and confirms OK), and `method="rmpflow"`
is in the enum. The semantic issue: `interpolate_trajectory` is called BEFORE
`setup_pick_place_controller`, with three hard-coded joint waypoints that
ignore actual cube positions. The template acknowledges this is "advisory" but
the structural success of the canonical depends on
`setup_pick_place_controller(belt_path=…)` which handles its own approach.
The interpolate_trajectory call has no observable effect on the pick. Tools
listed but doing nothing functional. **Severity: LOW — vestigial call.**

### Angle B — Prim-path consistency

**A32 / carousel-relative child paths — MED**
`workspace/templates/CP-NEW-kitting-station-6sku.json:17`
Six bin children at `/World/Carousel/Bin_N` with `position=[0.55, 0.0, 0.0]`
(local). Cubes at `/World/Carousel/Bin_N/Cube` with `position=[0.55, 0.0,
0.0525]`. But the template's `failure_modes[2]` admits "verify bin world z =
0.75 + 0.0 = 0.75" — the carousel is positioned at z=0.75 but its child Cube
prims' positions read absolute world coords (`0.0525` not `0.5025`). On real
USD execution, child-prim `position` is interpreted as LOCAL to the parent's
xform; cubes at local-z=0.0525 will end up at world-z=0.75+0.0525=0.8025 —
which is what they want. But `setup_pick_place_controller` reads world poses
via the controller's get_world_transform; positions match. Net: paths and z
arithmetic happen to align by coincidence-of-numbers, but the
**hard-coded LOCAL bin positions break after carousel rotation** — the
template's own failure_modes[0] flags this honestly. **Severity: MED.**

### Angle C — Code-template substitution coherence

**A38 / Jinja-like template substitution — CLEAN**
A38 `code_template` substitutes `{{reward_env.task_description}}` etc into
positional-context Python string literals. All substitutions match
role_defaults keys; no orphan references. The `code` and `code_template`
diverge slightly (full version has env_source_path file-write +
result-payload assertions; condensed has the bare pipeline) but both are
self-consistent. **OK.**

**A33 / opcua_bridge node_map substitution — BLOCKER (compounded)**
`workspace/templates/CP-NEW-machine-tender-door-interlock.json:298`
`code_template` line: `opcua_bridge_attach(url={{opcua_bridge.url}},
node_map={{"{{cnc_machine.path}}.door_closed:bool": {{cnc_machine.door_node}}},
…)` — nested Jinja substitution inside a dict literal will produce malformed
JSON or syntactically-broken Python after expansion (the outer braces double
as dict braces). A simple textual `.replace` substitution will yield
`node_map={"{cnc_machine.path}.door_closed:bool": ns=2;i=1}` — bare `ns=2;i=1`
is not a Python literal. **Severity: BLOCKER for template instantiation.**

## §4 Form-gate effectiveness

The enum-value + nested-item-required extensions (post-A30) **DO** close the
A21 (`joint_positions` per-item) and A30 (validate_assembly_constraint type
enum) classes of bugs. Lint now correctly accepts every schema-shaped call in
A31–A40 and would have caught A30's `cylindrical_fit` enum-miss.

What lint still cannot see:
- **Cross-call data flow** — A33's bridge_id must come from attach's return.
  The lint sees two independent schema-valid calls. Detecting this needs
  whole-program analysis of the `code` string.
- **Handler-side semantic conventions** — A37's `color_routing` only reading
  `Semantics_color/class` is a handler-implementation contract not exposed in
  the schema. Catching this needs cross-referencing handler code, not pydantic
  models.
- **Template-string Jinja correctness** — A33's nested `{{ }}` in a dict
  literal would render to malformed Python. Lint doesn't expand templates.
- **Semantic realism** (A35 conflation) — needs human judgement.

Effectiveness: ~85% of *new-style* bugs caught (compared to ~50% before the
A30 extension). Three deeper classes remain in the gap. Recommend lint v3 add:
(a) capture-required tool list (e.g. `*_bridge_attach` ⇒ bridge_id must be
captured before downstream `diagnose_*` / `*_detach`); (b) template-string
Python-expandability check.

## §6 Recommendation

1. **Patch A33** — capture `bridge = opcua_bridge_attach(…)`, use
   `bridge["bridge_id"]` in subsequent diagnose. Also rewrite
   `code_template`'s `opcua_bridge_attach` line to escape the nested dict
   literal (use a Python `{...}` placeholder built outside the template-string
   substitution, e.g. assign `_node_map = {…}` on a prior line).
2. **Patch A37** — switch `set_semantic_label(semantic_type="class")` so the
   existing `_cube_semantic_class` lookup finds the labels. Update template
   thoughts to clarify the routing key is the class data, not a per-material
   Semantics_* instance.
3. **Reclassify A35** — rename `task_id`/`goal` to reflect that the canonical
   exercises the *export plumbing* (finetune_stats → redact → export_finetune
   → export_dataset) not specifically HDF5 demo packaging. Or replace
   `export_dataset` with the actual GR00T HDF5-writer tool if one exists in the
   schema.
4. **Note A32 carousel-rotation honestly** in `goal` — the current pose
   captures index=0 only; full 6-cycle pick requires runtime carousel index
   updates not in the static template.
5. **Backlog A39's vestigial call** — either drop `interpolate_trajectory`
   from `tools_used` and `code`, or wire it into a real approach motion that
   uses its return value.

**Suggested lint v3 additions:**
- Capture-required pair rules (bridge_attach → diagnose/detach).
- Python-syntax check on `code_template` after stripped-Jinja substitution.
