## Top-3 findings (single sentence each)

1. **A113 joint_damping sweep result has no apply path** — the canonical
   sweeps `joint_damping` via `create_calibration_experiment(parameter="damping", ...)`,
   records `best_value` in val_result.calibrated_params, and writes it to
   the artifact JSON, but `set_drive_gains` only accepts `kp` and `kd` —
   there is no `joint_damping` kwarg on `SetDriveGainsArgs`, so the swept
   damping value is never applied to live UsdPhysics.PhysxJointAPI before
   the post-sweep effort regression check runs. **MEDIUM honesty** —
   failure_modes #6 mentions the parameter dispatch ('damping' routes to
   UsdPhysics.PhysxJointAPI damping inside the sweep) but does NOT
   acknowledge that the final-apply step is dropped; the value is stored
   for reporting only. Recommended patch: either (a) add explicit
   `set_attribute(prim_path=joint_path, attr_name="physxJoint:damping", value=best_damping)`
   loop after the kp/kd apply, OR (b) add `joint_damping_applied: false`
   field to role_defaults so consumers know the third axis is sweep-only.

2. **A111 hardware_spacemouse blocker uses graceful-fallback pattern but
   the structural gate accepts zero-length HDF5 as valid** — when
   `check_teleop_hardware(device="spacemouse")` returns `supported=False`,
   the canonical proceeds to `record_teleop_demo` which (per handler) creates
   structurally-valid zero-length HDF5 files in fallback mode, and the
   per-file `validate_teleop_demo` is documented to ACCEPT T=0 as a valid
   robomimic shape (failure_modes #6 + extension_notes). This means the
   form-gate passes on a hardware-absent host with empty data files —
   the gate is structural-only, not data-presence. **MEDIUM honesty** —
   failure_modes already documents this in entry #9 (the SUCCESS criterion
   gate). Recommended patch: add `requires_operator_input: true` flag in
   role_defaults so retrieval consumers querying for "real teleop demo
   corpus" understand the structural-gate-only contract; or add a
   `min_actions_per_demo: 1` field that the validate step can check when
   running outside fallback.

3. **A104 `count_prims_under_path(parent_path="/World")` overcounts** —
   the canonical's mass-conservation check intends to verify 4 brick prims
   are still on stage post-sweep, but `count_prims_under_path` with
   `parent_path="/World"` and no `type_filter`/`recursive` flag returns
   direct children of `/World` (DomeLight + Ground + Table + Baseplate +
   PickSensor + Franka + 4 bricks = 10 prims, NOT 4). The check is still
   useful as a "stage didn't lose prims" canary but doesn't bind to bricks
   specifically. **LOW honesty** — recommended patch: change to
   `count_prims_under_path(parent_path="/World", type_filter="Cube")` which
   filters to Cube prims (10 of which include Ground/Table/Baseplate/4
   bricks/etc.) — STILL overcounts. The cleanest fix is to count under a
   dedicated `/World/Bricks` parent Xform after re-parenting bricks; second
   cleanest is to drop the count_prims_under_path call entirely and use the
   per-brick `get_world_transform` loop's success count as the
   mass-conservation signal.
