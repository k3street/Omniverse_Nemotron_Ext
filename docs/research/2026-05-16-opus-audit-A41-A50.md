# Opus audit A41-A50 — 10 new canonicals

**Date**: 2026-05-16
**Scope**: A41 (CP-88) + 9 CP-NEW templates in `workspace/templates/`
**Depth**: 4 deep + 6 sampled (CP-88, vision-depalletize, multi-rate-sensor-fusion, ros2-multi-rate deep; others sampled)
**Schema source**: `service/isaac_assist_service/chat/tools/handlers/_models.py`
**Handler source**: `service/isaac_assist_service/chat/tools/handlers/ros2.py`, `sdg.py`, `scene_authoring.py`, `training.py`

## Verdict: YELLOW

Two of ten templates (A44 multi-rate-sensor-fusion, A47 ros2-multi-rate) have a **HIGH severity** semantic bug
that the lint extension cannot catch: per-sensor `topic`, `publish_rate`, `qos`, `hz` keys
in `configure_ros2_bridge(sensors=[...])` are **silently ignored** by the handler. The handler reads only
`type`, `prim_path`, `topic_name`, `frame_id`. This means:

- Templates that assert `expected_topic in ros2_list_topics()` will FAIL on `/camera/rgb` and `/joint_states`
  because the OG node's `inputs:topicName` is set to empty string when only `topic` is provided.
- The multi-rate claim in both templates is **fictitious** — all sensors share a single `OnPlaybackTick`
  clock; per-sensor rate is not wired anywhere in `_gen_configure_ros2_bridge`.

The remaining 8 templates pass deep + sampled review. Patch backlog: **6 items**.

## Top 3 findings

### F1 (HIGH) — A44 + A47: `configure_ros2_bridge` per-sensor key mismatch + ghost rate-control

**Affected**: `CP-NEW-multi-rate-sensor-fusion.json` (A44), `CP-NEW-ros2-multi-rate.json` (A47)

**Evidence**: `ros2.py:275-311` shows the handler reads only:
```python
stype = sensor.get("type", "camera")
prim_path = sensor.get("prim_path", "")
topic_name = sensor.get("topic_name", "")    # <-- NOT "topic"
frame_id = sensor.get("frame_id", "")
```
Templates pass `"topic": "/camera/rgb"`, `"publish_rate": 30`, `"qos": "sensor_data"`,
`"hz": 1000`. Every one of those keys is dropped silently.

**Consequence on A47**:
```python
active_topics = ros2_list_topics()
expected = {"/camera/rgb", "/joint_states", "/clock"}
missing = expected - set(active_topics)
if missing:
    raise RuntimeError(f"Multi-rate bridge missing topics: {missing}")
```
This will ALWAYS raise — the OG publisher nodes publish on empty-string topic names,
not on `/camera/rgb`/`/joint_states`. Self-check is a tripwire that always fires.

**Consequence on A44**: Less catastrophic (no assertion), but `expected_camera_hz=30`
and `expected_ft_hz=1000` in `verify_args` are unenforceable; both sensors will publish
at the OnPlaybackTick rate (physics step). Also `"type": "camera_rgb"` falls back to
the generic `ROS2PublishCamera_rgb` node class (line 289), which doesn't exist —
the agent would have used `"type": "camera"`.

**Patch**: Rename `topic`→`topic_name` in both templates; change `"camera_rgb"`→`"camera"`;
drop `publish_rate`/`qos`/`hz`/`force_torque` keys OR add a fallback comment noting the
single-rate publication (multi-rate is misleading in the canonical name).
Also: `"force_torque"` type is unsupported — `ROS2PublishForce_torque` class will not
register. Use the F/T sensor's own `publish_topic` (already set via `add_force_torque_sensor`)
and remove the FT entry from the bridge `sensors` list.

### F2 (MEDIUM) — A41 (CP-88): `openplc_runtime_attach` does not return a `bridge_id`

**Affected**: `CP-88.json` (OpenPLC bridge canonical)

**Evidence**: Template hard-codes `bridge_id="openplc_127.0.0.1_502"` in role_defaults
and passes it to `diagnose_modbus_bridge` + `modbus_tcp_bridge_detach`. But
`OpenplcRuntimeAttachArgs` (`_models.py:505-513`) takes no `bridge_id` argument, and the
underlying modbus handler returns a *generated* bridge_id (e.g.
`modbus_<host>_<port>_<pid_or_uuid>`). The template's `failure_modes` entry #2 itself
warns of this: "list_bridges returns the actual bridge_id assigned at attach time;
use that id for diagnose and detach, not a guessed value." Yet the code does exactly
the guessed-value pattern in the canonical body.

**Patch**: Capture the return value: `bridge_result = openplc_runtime_attach(...)`;
`bid = bridge_result["bridge_id"]`; pass `bid` to the two downstream calls. Mirror
the pattern used correctly in A45 (`CP-NEW-modbus-plc-fixture-io`).

### F3 (LOW-MEDIUM) — A50: `clone_envs` after `create_isaaclab_env(num_envs=8)` is double-cloning

**Affected**: `CP-NEW-curriculum-object-complexity.json`

**Evidence**: Template calls `create_isaaclab_env(task_name=..., num_envs=8, ...)` followed
by `clone_envs(source_path="/World/envs/env_0", num_envs=8, ...)`. `create_isaaclab_env`
already scaffolds env_0 through env_N internally (via the IsaacLab cloner); a second
`clone_envs` either no-ops (idempotent path) or doubles envs to 16. Either way the
launch_training call asserts num_envs=8, so this is at best dead code, at worst a
subtle off-by-N for VRAM accounting. The same anti-pattern appears in `CP-NEW-dr-curriculum-trainer`
(prior template) so this may be a propagation hazard.

**Patch**: Drop the standalone `clone_envs` call, OR call `create_isaaclab_env` with `num_envs=1`
and use `clone_envs(num_envs=8)` as the actual cloning step. Recommend the former (simpler).

## Other findings (LOW severity)

### F4 — A43 (vision-depalletize): `vision_detect_objects` ordering

`vision_detect_objects` is called BEFORE the camera is set as active viewport
(no `set_viewport_camera('/World/PalletCam')` precedes it). The template's
`failure_modes` notes this. Acceptable as drafted (drafted-2026-05-16) but
should be patched before function-gate.

### F5 — A42 (groot-redact-finetune): `format="openai"` may be rejected by handler

`export_finetune_data` (`_models.py:1814-1820`) accepts `format` as free-form string,
but the underlying recorder typically takes one of `{"openai", "anthropic", "ollama",
"alpaca"}`. The template uses `"openai"` which is correct; the failure_modes hedge
about `"openai_chat"` / `"openai_finetune"` is misleading — the handler does NOT
accept those. Edit failure_modes to remove the false alternatives.

### F6 — A49 (scene-sdg-full-pipeline): `create_sdg_pipeline` + `configure_coco_yolo_writer` ordering

Template calls `configure_coco_yolo_writer` BEFORE `create_sdg_pipeline`. `CreateSdgPipelineArgs`
takes `output_format`, `output_dir`, `annotators` — it can SUPERSEDE the writer config.
Verify with handler that both calls compose; if they conflict, switch order or drop
the explicit writer call. Marked LOW because both calls succeed independently.

### F7 — A44: `WristCamera` prim is never created

Template references `/World/Franka/panda_hand/WristCamera` in `configure_ros2_bridge`
sensors list, but never `create_prim`s a Camera at that path. The Franka robot
USD does not include a wrist-mount camera by default. The `failure_modes` notes
this. Code-template will fail until the prim is authored. Patch: add
`create_prim(prim_path="/World/Franka/panda_hand/WristCamera", prim_type="Camera")`
before the bridge call.

## Cross-template patterns observed

- **Bridge-validation triad** (list_bridges + diagnose_* + detach) is used consistently
  across A41/A45 — good convergence on the Phase 6/10 pattern.
- **DR axis count** A49 (4 axes) and A50 (5+ axes) extend the library's previous
  max of 3 axes — both correctly use `analyze_randomization` as the gate.
- **role_defaults vs hard-coded paths**: A45/A46/A50 use defaults consistently;
  A43 uses both (hard-coded `/World/ArrivalSensor` not in defaults) — minor.
- **HARD BLOCKER honesty**: A42/A46/A48/A49 all correctly mark `function-gate BLOCKED`
  with specific reason (HDF5 data, GR00T weights, omni.replicator). Good.

## Recommendation

1. Land F1 first (A44 + A47) — these are silently-broken self-checks that will
   manifest as `RuntimeError: missing topics` on first execution, blocking
   function-gate progression for the entire multi-rate-ros2 lineage.
2. F2 (A41) is a 5-line edit; bundle with F1.
3. F3 (A50) and F4/F5/F6/F7 are cleanup; can wait for the next QC round.
4. Consider adding a lint rule for `configure_ros2_bridge.sensors[].topic` →
   suggest `topic_name`, and reject unknown sensor types (force_torque,
   camera_rgb) to catch F1 cross-call mismatches automatically.
