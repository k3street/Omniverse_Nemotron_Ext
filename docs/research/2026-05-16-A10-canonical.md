## ¬ß4 What's Distinct About Navigate vs A1-A9

A1-A9 patterns: pick_place√ó3, sort√ó2, other√ó2 (turn-faucet, ros2-bridge), reorient√ó1, rl-train√ó1.
None chain occupancy-map generation into navigation.

**Navigate pattern distinctives:**

1. **No manipulation** ‚Äî the workpiece is the robot itself; `simulate_args.cube_path` and
   `verify_args.cube_path` both point to the AMR, not a manipulated object.

2. **Map pipeline** ‚Äî `generate_occupancy_map` + `export_nav2_map` are environment-query tools
   that run before simulation. They produce side-effect artifacts (PGM/YAML files) rather than
   scene prims. This is a fundamentally different execution model from all A1-A9.

3. **Planner as kwarg** ‚Äî `navigate_to(planner="astar")` explicitly uses the grid planner so the
   path is non-trivial (routes around 4 shelf obstacles). Pick-place canonicals use cuRobo trajectory
   planner via `setup_pick_place_controller`; navigate has its own simpler A* pipeline.

4. **`motion_controllers` tag** ‚Äî navigate does not use motion-planning tools in
   `MOTION_PLANNING_TOOL_PREFIXES`, so no motion_controllers WARN is triggered. Field is included
   voluntarily for controller-filtered retrieval completeness, with both `verified` and `failed` empty
   (no runs yet) and `untested` listing controllers that could theoretically wrap the AMR.
